# 综合运营报表最终呈现方案

## 1. 页面定位

监控报表现在是一个按筛选条件统计的**综合运营报表页**，不是实时监控页。

用户进入后先选择日期、机器人和呼叫方向，然后从上到下查看三类内容：

1. 通话统计
2. 业务与流程分析
3. 工具调用

页面不再使用多个一级 Tab，也不再放通话明细；通话明细统一去左侧“通话记录”菜单查看。

---

## 2. 顶部筛选

| 筛选项 | 展示方式 | 影响范围 | 值来源 |
| --- | --- | --- | --- |
| 时间范围 | 今日、昨日、本周、上周、本月、上月快捷按钮 | 全部报表 | 通话开始时间、流程事件时间、工具事件时间 |
| 机器人 | 多选下拉 | 全部报表 | botId |
| 呼叫方向 | 下拉 | 通话、流程、工具统计 | callDirection |
| 导出 | 按钮 | 导出当前筛选后的报表 | 当前筛选参数 |
| 刷新 | 按钮 | 重新拉取当前筛选口径下的数据 | 当前筛选参数 |

当前前端 demo 中，日期筛选会驱动 mock 数据变化；机器人和呼叫方向先保留交互形态，接后端时作为查询参数。

## 3. 通话统计

### 3.1 顶部指标卡

顶部指标卡分两行展示，避免一行挤满导致可读性差。

| 指标 | 含义 | 计算公式 |
| --- | --- | --- |
| 通话总量 | 当前筛选范围内所有呼入和外呼通话数 | `count(call_id)` |
| 接通率 | 有接通的通话占比 | `接通通话数 / 通话总量` |
| 有效通话率 | 超过有效时长阈值的通话占比 | `count(duration > 15s) / 通话总量` |
| 平均通话时长 | 接通通话的平均时长 | `avg(duration)` |
| 转人工量 | 当前筛选范围内触发人工转接的次数 | `count(transfer_event)` |
| 重复呼入客户 | 当前筛选范围内多次呼入的客户数 | `count(distinct phone where call_count > 1)` |
| 用户主动挂断量 | 用户侧主动结束通话的数量 | `count(user_hangup_event)` |

转人工量、重复呼入客户、用户主动挂断量都展示“数量 + 占比”，占比分母统一用当前筛选范围内的通话总量。

### 3.2 图表和列表

| 模块 | 展示内容 | 用户价值 |
| --- | --- | --- |
| 通话量趋势 | 按小时或按日用折线图展示当前呼叫方向下的主通话量和有效结果量 | 看业务量波动、高峰和有效处理变化 |
| 挂断 / 未接通原因 | 展示正常结束、用户主动挂断、接通前放弃、超时、异常中断、无人接听等 | 判断通话损失主要来自哪里 |
| 通话时长分布 | 展示不同时长区间的数量和占比 | 识别秒挂、短通话、超长通话 |
| 地域分析 | 展示各地域呼叫量和接通率 | 判断地域差异，避免把流程完成率误归因到地域 |
| 时段热力图 | 展示一周七天、每天 24 小时的通话量 | 判断高峰时段 |
| 重复呼入 TOP 客户 | 展示多次呼入客户和最近结果 | 发现未解决或反复来电客户 |
| 短通话样本 | 展示短通话 ID、时长、原因和最近节点 | 快速抽查异常样本 |

重复呼入 TOP 客户和短通话样本都需要支持搜索、排序、分页和每页条数切换；筛选过窄时展示空状态。
通话量趋势按顶部“呼叫方向”切换口径：

| 呼叫方向 | 蓝线 | 绿线 | 原因 |
| --- | --- | --- | --- |
| 全部呼叫 | 总通话数 | 有效通话数 | 看整体量级和有效处理量 |
| 呼入 | 呼入量 | 有效接待量 | 呼入是用户主动来电，不用“接通量”衡量 |
| 外呼 | 外呼量 | 接通量 | 外呼才需要重点看拨打后的接通结果 |

## 4. 业务与流程分析

这一块拆成两个表：**Topic 主题分析表**和**Flow 流程分析表**。

拆开的原因：Topic 是“用户在聊什么”，Flow 是“流程执行得怎么样”。两者不能混在一张表里，否则会误导用户把转人工、主动放弃等结果归因到某个主题上。

### 4.1 Topic 主题分析表

Topic 只表达需求分布，不展示流程完成、转人工、主动放弃等结果指标。

| 字段 | 含义 | 计算公式 |
| --- | --- | --- |
| 主题 | 用户需求主题名称 | `topic_name` |
| 命中通话数 | 有多少通话出现过该主题，同一通话出现多次只算一次 | `count(distinct call_id where topic_name = 当前主题)` |
| 主题占比 | 该主题覆盖了多少通话 | `命中通话数 / 总通话数` |
| 首次主题通话数 | 该主题是通话第一个主题的通话数 | `count(distinct call_id where first_topic = 当前主题)` |
| 首次主题占比 | 该主题作为首个问题的占比 | `首次主题通话数 / 总通话数` |

不展示：趋势、代表原话、最高卡点步骤、流程完成量、转人工量、主动放弃量。

交互要求：支持主题名称搜索、点击表头排序、分页和每页条数切换。

### 4.2 Flow 流程分析表

Flow 主表只展示流程级统计，并支持展开查看 Step 和边。

| 字段 | 含义 | 计算公式 |
| --- | --- | --- |
| Flow 名称 | 流程名称 | `flow_name` |
| 机器人 | 该 Flow 所属机器人 | `bot_name` |
| 进入量 | 进入该 Flow 的次数 | `count(flow_start_event)` |
| 完成率 | 完成该 Flow 的占比 | `count(flow_completed_event) / 进入量` |
| 平均流程时长 | 从进入 Flow 到离开 Flow 的平均时长 | `avg(flow_exit_time - flow_start_time)` |
| 明细 | 展开查看 Step 明细和边 / 分支明细 | 前端交互 |

不展示：到达结束节点量、中途离开量、下一步到达量、下一步到达率。

交互要求：支持 Flow 名称 / 机器人搜索、点击表头排序、分页、每页条数切换；同一时间只展开一个 Flow，避免页面过长。

### 4.3 Flow 展开：Step 明细

| 字段 | 含义 | 计算公式 |
| --- | --- | --- |
| Step 名称 | 流程步骤名称 | `step_name` |
| Step 类型 | 开始、采集、工具、结束等类型 | `step_type` |
| 进入量 | 进入该 Step 的次数 | `count(step_enter_event)` |
| 平均停留时长 | 用户在该 Step 的平均停留时间 | `avg(step_exit_time - step_enter_time)` |
| 转人工量 | 在该 Step 触发转人工的次数 | `count(transfer_event where step_id = 当前 Step)` |
| 用户挂断量 | 在该 Step 用户主动挂断的次数 | `count(user_hangup_event where step_id = 当前 Step)` |

### 4.4 Flow 展开：边 / 分支明细

| 字段 | 含义 | 计算公式 |
| --- | --- | --- |
| 起点 Step | 连线来源步骤 | `from_step` |
| 终点 Step | 连线目标步骤 | `to_step` |
| 分支类型 | 普通连线、条件分支或大模型分支 | `edge_type` |
| 条件说明 | 该分支的触发条件说明 | `condition_text` |
| 命中次数 | 该边被走过的次数 | `count(edge_hit_event)` |
| 命中占比 | 该边在起点 Step 下的占比 | `命中次数 / 起点 Step 进入量` |

## 5. 工具调用

工具区块只保留**工具调用报表**。转人工整体数据已放在通话统计顶部指标中。

| 字段 | 含义 | 计算公式 |
| --- | --- | --- |
| 工具名称 | 被调用的工具或函数名称 | `tool_name` |
| 关联机器人 | 使用该工具的机器人数量 | `count(distinct bot_id)` |
| 类型 | API、SMS、TRANSFER、FUNCTION 等类型 | `tool_type` |
| 调用次数 | 当前筛选范围内工具调用总次数 | `count(tool_call_event)` |
| 成功率 | 工具调用成功占比 | `success_count / call_count` |
| 平均耗时 | 工具调用平均响应耗时 | `avg(latency_ms)` |
| 超时次数 | 工具调用超时次数 | `count(timeout_event)` |
| 直接播报次数 | 结果直接播报给用户的次数 | `count(direct_play_event)` |
| 直接播报成功率 | 直接播报成功占比 | `direct_play_success_count / direct_play_count` |
| 调模型回复次数 | 工具结果转交模型整理回复的次数 | `count(model_reply_event)` |
| 节省模型次数 | 工具结果无需再次调模型的次数 | `saved_model_calls` |
| 失败原因 TOP1 | 当前工具最主要的失败原因 | `top(failure_reason)` |

工具调用报表里不要把“关联机器人”“直接播报成功率”“节省模型次数”等副信息塞在同一个单元格里，要拆成独立列展示，便于排序和横向对比。

交互要求：支持工具名称、类型、失败原因搜索；支持排序、分页和每页条数切换。

## 6. 报表通用交互和状态

即使是前端 demo，也要具备真实报表的基本交互形态：

| 能力 | 适用范围 | 说明 |
| --- | --- | --- |
| 加载态 | 全页和各模块 | 筛选切换、刷新时展示骨架，不让页面瞬间跳变 |
| 搜索 | Topic、Flow、重复呼入、短通话、工具 | 用于快速定位具体主题、流程、客户或工具 |
| 排序 | 所有主要表格 | 表头点击排序，默认按最常用关注指标排序 |
| 分页 | 所有可能增长的列表 | 显示总数、当前范围、上一页、下一页 |
| 每页条数 | 所有可能增长的列表 | 支持 5 / 10 / 20 条，模拟真实后台表格 |
| 空状态 | 所有表格 | 当前筛选或搜索无结果时给出明确说明 |
| 导出反馈 | 全页 | Demo 中用提示反馈；真实系统接导出任务 |

## 8. 验收标准

- 页面只有一个综合运营报表页面，没有一级 Tab 和二级 Tab 导航。
- 顶部日期筛选影响全部保留模块的数据。
- 页面展示刷新和导出按钮。
- 主要表格支持搜索、排序、分页、每页条数、加载态和空状态。
- 通话明细不在本页展示，只跳转到“通话记录”菜单。
- Topic 表只看需求分布，不混入流程完成、转人工、主动放弃。
- Flow 表只看流程执行，主表可展开 Step 明细和边 / 分支明细。
- 不展示实时监控、告警中心、报表订阅、排队、空闲坐席等监控或坐席系统指标。
